# 营小助配置中心需求规格说明
> 状态：已确认
> 创建日期：2026-03-09
> 所属里程碑：无

## 1. 概述

### 1.1 功能简介
营小助配置中心用于辅助银行内部运营与审核人员，配置中心分为两块能力：智能提示（哨兵）与智能作业。智能提示侧重监管临时要求传达、事中风险提醒与新员工作业指引；智能作业侧重页面识别、数据获取、自动注入与执行控制。系统通过统一配置中心管理触发条件、规则及其提示配置、作业场景与字段配置，插件在作业页面执行事件监控、提示渲染与自动处理，减少重复录入成本并提升作业一致性；差异高亮由第三方应用提供。

```
    │   ├── 通用能力中心
    │   │   ├── ✅ 页面资源管理模块【静态配置中心】（监控网站/网页/元素配置、页面识别规则）
    │   │   │   └── 【对外提供能力：页面元素元数据查询、元素选择器映射，两个子系统共享只读权限】
    │   │   ├── 接口定义管理
    │   │   ├── 预处理器管理
    │   │   └── 资源共享管理

    │   ├── 哨兵配置后台

    │   └── 哨兵浏览器插件
    │       ├── 页面识别引擎
    │       ├── 规则匹配执行器
    │       ├── 提示渲染组件
    │       └── ✅ 事件产出模块【对外输出：页面上下文、当前页面**所有已配置元素**的完整元数据（逻辑名、选择器、当前值）、触发规则信息】

        │   ├── 组件管理模块（录入/赋值/接口请求/取值等基础组件）

            └── 悬浮按钮交互组件
```

#### 模块依赖说明
页面资源能力采用「静态配置统一管理 + 运行时解析唯一入口」架构，避免重复建设与数据不一致：
1. **静态配置层**：页面、元素、选择器、页面识别规则仅需在公共支撑层配置一次，两个子系统共享只读权限，配置入口唯一，全局生效
2. **运行时执行层**：页面识别、元素解析、取值逻辑仅由哨兵插件统一执行一次，避免多插件重复解析页面导致的性能损耗与结果不一致
3. **事件共享机制**：哨兵插件产出的触发事件包含当前页面所有已配置元素的完整元数据（逻辑名、选择器、当前值等），智能作业无论需要操作多少个元素都可以直接从事件中获取
4. **智能作业零页面依赖**：智能作业子系统完全不需要自行实现页面解析、元素查询相关能力，也没有页面配置权限，所有页面相关能力全部复用公共层配置与哨兵插件的解析结果
5. **变更一处全量生效**：当页面结构变化时，仅需在公共层更新一次元素选择器，所有引用该元素的规则、场景自动生效，无需重复修改

### 1.4 术语
- 组件：最小可执行单元，例如取值、判断、赋值、格式转换等操作。
- 场景：由一个或多个组件组成的完整执行流程；一个场景最少包含一个组件。
- 场景编排：用户从组件库拖拽组件并建立连线关系，形成完整场景流程的配置方式。
- 哨兵插件：独立页面监控插件，负责监听页面变化并按触发条件产出事件。
- 监控网站：被监控的站点维度，通常对应业务系统域名或统一入口。
- 监控网页：监控网站下的具体页面，通过 URL 规则（精确/通配/正则）识别。
- 本页元素：在监控网页下维护的页面元素逻辑名与选择器绑定，用于规则引用。
- 条件：最小判断单元，由左值、运算符、右值组成。
- 规则模板：不绑定具体页面的规则抽象，可通过占位元素映射复用到目标页面。
- 预处理器：在条件比较前对元素值或接口值做转换的能力单元，可内置或自定义。
- 提示配置：依附于规则存在的提示定义，包含提示类型、提示内容、处理方式、适用范围等属性，不作为独立配置对象存在。
- 提示类型：规则命中后的处理形态，首期包含静默、浮窗、外部通知三类；静默不显示浮窗但正常记录日志，浮窗在页面内展示提示内容，外部通知表示通过站外通知方式处理，具体对接方案不在本文展开。
- 提示标签：用于标识提示的业务语义与检索维度，如监管临时要求、事中风险、新员工作业指导等；标签不作为提示类型，也不直接决定运行时处理优先级。
- 规则触发日志：规则命中后的运行记录，用于追溯规则何时、因何、向谁触发，以及当次命中的字段匹配情况与处理结果。
- 分行管理人员：负责本分行规则维护、规则启停、时效管理与审计查看的业务管理角色。

### 1.5 配置中心设计目标
- 统一配置入口：将页面管理、场景编排、触发协同、执行控制、审计追溯统一到营小助配置中心内，避免多系统割裂维护。
- 低门槛配置：业务人员以可视化方式完成大多数配置，脚本能力仅用于复杂场景兜底。
- 安全可控发布：配置变更需可追踪、可回滚、可分级放量，避免一次发布影响全部作业页面。
- 运行稳定优先：在页面结构变化、网络抖动、配置异常时，系统默认降级到可继续作业的路径。

### 1.6 智能提示业务背景与定位
- 监管临时要求承接：针对监管机构或上级单位发布的临时性作业要求，在业务系统短期无法完成产品化改造时，智能提示作为过渡性控制措施，向一线人员提供明确操作提醒与处理口径。
- 事中风险防控：针对办理过程中的关键风险点（资料缺失、步骤遗漏、校验异常、时效临近等），智能提示提供实时风险提醒，降低违规与差错概率。
- 新员工作业引导：针对新员工高频办理事项，智能提示提供步骤化指引与关键注意事项，缩短上手周期并提升作业一致性。
- 生命周期治理原则：智能提示定位为“轻量、快速、可下线”的辅助能力；当正式系统能力上线并稳定运行后，对应临时提示应按流程停用，避免双轨规则并存。

### 1.7 分期上线策略
- 一期支持能力拆分上线，允许“智能提示”与“智能作业”二选一先行，不要求同步上线。
- 当前一期优先建设公共支撑层核心能力：页面资源管理（页面元素配置、页面识别规则）、系统管理、通用能力中心，作为全局共享基础资源。
- 同时一期交付“智能提示（哨兵）”完整运行时能力：页面识别、元素解析、规则引擎、事件产出、提示下发全链路能力。
- 智能作业核心能力（场景编排、自动录入等）按后续里程碑推进，智能作业无需重复建设页面相关能力，完全复用公共层的页面配置与智能提示模块的触发事件执行。
- 首期项目成功标准采用“合规风险 + 效率质量”双目标，不以单一维度评估上线成效。

## 2. 用户故事与场景

### 2.1 目标用户
- 银行内部运营人员
- 银行内部审核人员
- 字段配置维护人员（业务侧）
- 页面配置维护人员（业务侧/实施侧）
- 分行管理人员（负责本分行规则维护、规则启停、时效管理与审计查看）
- 总行管理人员（负责全行强控规则下发、全局配置管理与审计查看）

### 2.2 用户故事
- 作为运营人员，我希望从接口获取业务数据并自动注入当前页面，以便减少重复录入时间并降低录入错误率。
- 作为审核人员，我希望插件能稳定获取页面字段信息并交由第三方应用展示差异，以便更快识别风险项并完成审核。
- 作为业务配置人员，我希望在后台系统中直接维护字段配置并可追踪版本，以便快速调整规则且可审计。
- 作为业务配置人员，我希望通过拖拽方式组合组件形成复杂注入场景，以便无需编写脚本也能快速完成配置。
- 作为业务配置人员，我希望通过场景编排把组件连成完整流程，以便可视化完成场景开发。
- 作为页面配置维护人员，我希望管理可执行页面及页面识别规则，以便将注入场景准确绑定到目标页面。
- 作为运营管理人员，我希望在临时监管要求下快速上线作业提示，以便在系统改造前保障执行一致性与合规性。
- 作为风险管理人员，我希望在业务办理过程中触发风险提示，以便及时纠偏并降低事中风险事件。
- 作为新员工，我希望在关键办理节点收到标准化操作指引，以便按规范完成业务办理。

### 2.3 使用场景
- 场景1：运营录入场景
前置条件：用户已登录作业系统，插件已安装并启用，后台已配置录入场景。
期望结果：系统检查页面场景绑定并注入悬浮按钮；按场景执行模式完成自动执行、预览确认或用户点击触发执行。

- 场景2：审核比对场景
前置条件：页面字段可被插件读取，第三方差异应用可用。
期望结果：插件提取字段信息并交由第三方应用处理差异高亮，审核人员可快速定位差异项并处理。

- 场景3：异常中断场景
前置条件：网络抖动、页面结构变化或字段缺失导致自动流程无法完整执行。
期望结果：系统给出错误提示并自动降级至手动模式，支持用户一键重试，不影响用户继续手工作业。

- 场景4：监管临时要求提示场景
前置条件：监管机构发布临时性办理要求，正式系统能力尚未完成改造；分行管理人员已配置提示内容并启用。
期望结果：目标页面在关键办理节点展示规范提示，指导一线人员按临时要求执行；后续由分行管理人员手工停用。

- 场景5：事中风险提示场景
前置条件：风险规则已在哨兵后台配置并启用。
期望结果：用户办理过程中命中风险规则时，系统实时提示风险原因与处理建议，并记录触发日志。

- 场景6：新员工作业指导场景
前置条件：新员工作业指导规则已配置并绑定目标页面。
期望结果：新员工在关键节点看到步骤化指引与注意事项，减少漏项并提升操作一致性。

## 3. 功能需求

### 3.1 输入
- 组织与人员基础数据（用户、部门树、用户部门归属；由外部系统同步）
- 员工身份标签数据（含“新员工”标识；来源 HR 系统同步）
- 监控网站与监控网页配置（URL 匹配模式、归属部门、可见性）
- 本页元素配置（元素名、选择器、元素类型/备注）
- 哨兵规则配置（条件列表、条件逻辑、提示配置、生效范围、有效期、状态）
- 规则模板配置（占位元素、模板条件、模板提示）
- 接口定义配置（请求参数、入参来源、响应取值路径、超时重试）
- 预处理器配置（内置/自定义、参数 schema、脚本与分类）
- 规则提示配置（提示类型、提示内容、处理方式、适用范围）
- 监管临时要求内容（由分行管理人员维护）
- 事中风险规则（由风控/业务管理人员维护）
- 新员工作业指导内容（由培训/业务管理人员维护）
- 接口返回的业务数据（文本、数值、日期等）
- 目标系统页面可写入字段位置
- 页面管理配置（页面编码、URL匹配条件、启停状态）
- 场景配置清单（由业务人员维护）
- 场景组件配置（组件类型、执行顺序、参数）
- 场景级悬浮按钮配置（位置、显示信息、显示条件、交互行为）
- 哨兵触发配置（由哨兵后台维护的触发条件与触发规则标识）
- 用户触发动作（手动触发或自动触发）

### 3.2 输出
- 规则拉取结果（按 URL、用户、部门匹配后的生效规则列表）
- 规则命中详情（命中规则、命中条件、取值来源、失败原因）
- 提示处理结果（处理类型、处理时间、处理结果、命中规则）
- 规则触发日志（规则 ID、提示类型、触发原因、触发页面、操作人、分行、处理结果）
- 规则状态变更记录（操作类型、操作人、操作时间、操作原因、作用范围）
- 自动录入完成状态（成功、部分成功、失败）
- 页面字段提取结果（供第三方差异应用使用）
- 预览确认结果（确认执行、取消执行）
- 页面命中结果（命中页面ID、命中场景版本）
- 失败时的错误提示与可恢复建议

### 3.3 核心行为描述
- 用户进入作业页面后，插件可被识别并处于可用状态。
- 用户进入页面后，哨兵插件独立完成页面识别、元素解析、规则匹配全流程。
- 哨兵插件根据已启用规则及其提示配置判断是否命中提示条件，命中后按提示类型执行对应处理：静默记录、浮窗展示或外部通知。
- 临时监管类提示处理完成后，用户可继续办理业务流程；提示仅用于作业指导与风险提醒，不阻断业务办理。
- 哨兵插件按其后台配置监听页面变化并产出触发事件（包含页面ID、元素值、匹配规则等完整上下文）。
- 智能作业插件无需自行实现页面识别能力，仅需消费哨兵插件产出的触发事件，判定是否执行对应命中场景。
- 若场景匹配成功，智能作业插件在页面注入悬浮按钮作为统一执行入口；若不存在匹配场景则不注入入口。
- 用户手动触发“自动录入”，或由外部系统调用智能作业插件 API 触发自动录入后，系统获取接口数据并按映射关系填充目标系统字段。
- 自动录入过程中，系统应保证仅操作当前业务记录对应字段，不跨记录写入。
- 自动录入前，系统提供预览模式，用户确认录入字段后才继续执行写入。
- 用户触发差异流程后，系统负责提取页面字段信息并输出给第三方差异应用。
- 对缺失、格式不合法或不可读字段，系统应标记为异常项并提示人工处理。
- 自动录入或字段提取失败时，系统应停止当前自动流程并给出可理解提示，自动降级为手工处理并支持重试。
- 分行管理人员可按本分行要求手工停用临时规则，停用后系统不再展示该规则产生的提示并保留审计记录。

### 3.4 页面元素字段管理
**能力归属**：页面资源配置属于公共支撑层静态配置能力，两个子系统共享只读权限；运行时页面识别、元素解析能力归属智能提示（哨兵）模块，智能作业模块无需重复建设，完全复用哨兵模块的页面识别结果与事件触发能力。

#### 3.4.1 配置流程
页面元素采用三级层级配置流程：
1. **第一步：配置监控网站**：录入业务系统名称、域名、归属部门、可见范围，完成站点级注册
2. **第二步：配置监控网页**：在站点下添加具体页面，配置页面编码（对应业务系统API返回的唯一页面标识）、URL匹配规则（精确/通配/正则三种模式）、页面描述、生效状态
3. **第三步：配置本页元素**：在具体页面下添加字段元素，配置元素逻辑名（业务侧易懂的名称，如"客户身份证号"）、CSS/XPath选择器、元素类型（文本/下拉/单选/多选/日期等）、备注说明、是否必填等属性

#### 3.4.2 使用逻辑
- 配置完成的页面元素可直接被同页面下的规则条件、场景组件、字段映射引用，引用时只需选择元素逻辑名无需关心具体选择器
- 规则模板中的占位元素可通过映射到目标页面的本页元素，实现跨页面复用
- 元素值支持配置预处理链，在参与规则比较或字段赋值前统一做格式转换、数据清洗等操作

#### 3.4.3 版本与权限管理
- 页面元素跟随所属页面统一版本管理，每次修改发布生成新版本，保留历史版本可追溯回滚
- 权限跟随所属站点/页面的权限体系：配置人员仅可编辑归属本部门的页面元素，其他部门共享的元素仅可查看引用，不可修改
- 支持元素跨部门共享，共享模式为只读复用，其他部门可Fork复制到本部门页面下独立维护，不影响原元素
- 元素被规则/场景引用时会建立引用关系，被引用的元素不可直接删除，需先解除所有引用后才可删除，避免配置失效
- 元素被引用后仍可修改，修改后自动生成新版本；引用方可自行选择是否升级到新版本，升级提示机制纳入后期迭代

### 3.5 哨兵规则模型与执行设计
- 规则绑定模型采用“监控网站 -> 监控网页 -> 本页元素 -> 规则”的层级结构，规则通过 `webpageId` 绑定具体页面。
- 规则由条件集合组成，单条规则条件数量为 `1~10`，条件逻辑支持 `AND/OR`，条件左右值来源支持”元素/固定值/接口”三类。
- 条件左右值均可配置预处理链，按步骤顺序执行后再参与比较；固定值默认不做预处理。
- 预处理器执行失败时，整个规则判定失败，按”失败不触发”策略处理并记录日志。
- 规则生效范围按”部门及其子部门”语义计算，子部门递归包含所有子孙部门；用户仅归属一个部门，由外部系统同步。
- 规则状态至少包含：草稿、已发布、已停用、已过期；到达过期时间后自动失效。
- 支持规则模板跨页面复用：先选模板，再做占位元素映射，生成页面级具体规则。
- 规则模板映射时必须完成所有占位元素的映射，未完成映射时系统禁止保存并提示"模板映射不完整，请为所有占位元素指定目标页面元素"。
- 提示内容采用富文本存储并支持占位符（元素值、接口值）运行时替换，替换值需做转义与安全过滤。
- 提示类型首期固定为三类：静默、浮窗、外部通知；三种类型命中后均必须写入规则触发日志，其中静默不展示浮窗，浮窗在页面内展示，外部通知按配置执行站外通知。
- 资源共享采用统一模型：归属部门、可见性（私有/指定部门共享/全局）、共享范围（部门及子部门可见）。
- 跨部门共享资源默认只读复用，支持复制（Fork）为本部门资源后再独立维护。
- 规则编辑/发布前需校验引用资源（监控网页、接口、预处理器、模板）权限；引用失效时禁止发布。
- 规则执行支持失败策略与降级策略：接口失败、预处理失败、资源异常时按”失败不触发”策略处理并记录日志。
- 插件拉取规则时应携带 `userId/deptId/URL`，服务端返回当前用户在当前页面有效期内的规则集。
- 接口来源值强制由服务端统一代调并返回，避免插件暴露敏感信息并统一配置治理。
- 接口调用超时时间在接口定义中统一配置，建议默认值为5秒；规则引用时不支持覆盖超时配置。

### 3.6 业务规则与约束
- 插件只服务银行内部已授权用户与指定业务系统。
- 自动录入前后，用户可见并可复核关键结果。
- 差异高亮能力由第三方应用提供，本系统不承载高亮渲染逻辑。
- 在插件不可用或页面不兼容时，用户仍可按原有手工流程继续作业。
- 插件安装与运行不得破坏现有业务页面核心功能。
- 字段配置维护应面向业务人员可用，操作应简单、可理解、可回退。
- 页面管理应支持页面注册、页面识别条件维护、场景绑定、启停控制与版本记录。
- 插件执行前需先完成页面识别，仅在命中已启用页面配置时才执行对应场景。
- 页面识别优先使用页面编码；页面编码由插件通过 API 获取。
- 当页面编码不可用或未命中时，使用 URL 作为保底识别条件。
- 自动录入与字段提取均需支持手动触发和自动触发两种模式。
- 场景触发条件配置由哨兵后台统一维护，智能作业后台不重复维护触发条件。
- 自动触发由外部系统实现，本系统通过插件 API 提供执行入口。
- 智能提示需区分“临时过渡类提示”与“长期运营类提示”，并支持按分行进行启停管理。
- 对临时过渡类提示由各分行管理人员结合业务实际手工评估停用时机；正式系统能力上线后应及时停用对应提示并保留审计记录。
- 支持总行下发“全行强控规则”，强控规则可直接对全行启用，分行无停用权限。
- 智能提示内容应使用统一规范术语与标准文案模板，避免跨分行提示口径不一致。
- 同一页面同一时刻命中多个提示时，规则可同时生效；系统应按优先级进行分层排序展示，避免提示冲突与重复干扰。
- 智能提示触发默认采用”同一页面、同一规则单次触发”机制，避免同一规则在稳定条件下重复打扰用户。
- 静默类型命中后不展示页面浮窗，但必须写入规则触发日志。
- 浮窗类型命中后在页面内展示提示内容，并必须写入规则触发日志；首期浮窗采用串行展示（逐条弹窗），用户关闭当前提示后自动展示下一条；合并浮窗展示方式纳入后续迭代优化。
- 外部通知类型命中后按配置执行外部通知处理，且必须写入规则触发日志。
- 默认同一页面同一规则仅触发一次；若规则中的某个条件被标记为“可重复触发”，则当该条件涉及字段的值发生变化时，该规则重新进入待触发队列，并允许再次判定与展示。
- 底层模型允许多个条件标记为“可重复触发”，但首期产品能力仅开放单个条件配置，待业务语义明确后再开放多条件场景。
- 智能提示触发时机首期仅支持“页面加载后触发、字段变化时触发”，不支持提交前触发。
- 插件采用常驻监听模式，在浏览器中持续运行；仅在命中页面或命中页面变化事件时进入实际工作状态。
- 插件通过 DOM 监听直接感知页面变化，不依赖业务系统事件总线或页面内回调。
- 规则触发采用混合模式：页面首次加载时对当前有效规则执行一次判定；字段变化后，仅对尚未命中的规则继续判定，已命中规则默认过滤。
- 对于已命中的规则，默认从待触发队列中过滤；若某个被标记为“可重复触发”的条件所涉及字段值发生变化，则该规则重新进入待触发队列。
- 条件字段是否发生变化以 DOM 实际取值的新旧值比较结果为准，不以 DOM 节点重绘、组件重渲染或事件触发本身作为变化依据。
- 插件配置拉取采用“进页面拉取一次”的模式，不在页面驻留期间做高频主动轮询；页面内后续规则触发基于本地已拉取配置执行。
- 接口调用失败、预处理失败、判定超时等执行异常场景统一采用“失败不触发”策略。
- 场景执行模式需支持：自动执行、预览确认、手动触发（无自动操作，等待用户点击悬浮按钮）。
- 每个场景都支持个性化悬浮按钮配置。
- 每个场景可根据业务需求独立配置执行模式，不同场景的执行模式可以不同。
- 场景级悬浮按钮配置至少包括：位置（四角+偏移）、显示信息（标题/状态文案）、显示条件、交互行为。
- 由于为内部系统，首期对外部系统调用插件 API 的鉴权采用简化方案，后续按安全要求迭代增强。
- 场景配置采用“草稿保存不生效、发布后生效”的治理模式，同时必须保留版本记录用于追溯与回滚。
- 复杂字段注入应优先支持拖拽式可视化编排（节点/步骤组合），脚本能力作为高级兜底能力。
- 场景由组件编排形成，且场景最少包含一个组件。
- 场景编排应基于组件库完成，用户可拖拽组件并通过连线定义执行关系。
- 场景编排首期允许分支流程。
- 场景编排首期不支持回环流程，回环能力后续迭代支持。
- 首期组件范围固定为：录入组件、赋值组件、接口请求组件、获取字段值组件。
- 首期不在上述清单内的组件能力不纳入交付范围，后续按迭代扩展。
- 字段映射除直接映射外，还需支持脚本化处理能力，以满足拼接、格式转换等复杂字段处理场景。
- 可视化编排至少支持：取值、判断、赋值、拼接、格式转换、默认值、预览调试。
- 可视化编排与脚本配置均纳入统一场景版本管理与审计记录。
- 脚本执行需设置超时与失败兜底，超时或异常时返回可理解错误并支持回退默认映射或手工处理。
- 脚本沙箱隔离能力首期暂不实现，后续迭代补充。
- 同一页面允许配置多个录入场景；首期默认”单场景模式”，仅执行命中的最高优先级场景。
- 场景优先级由配置人员手动指定数值（建议范围1-999，数值越大优先级越高）；当多个场景优先级相同时，按场景创建时间倒序排序（最新创建的优先）。
- 多场景组合执行能力（如串行执行）不纳入首期，后续迭代再评估。

### 3.7 配置对象与关系设计
- 配置中心管理的核心对象包括：页面、规则、规则提示配置、场景、组件、字段映射、悬浮按钮、版本、发布记录、审计记录。
- 页面是配置绑定的入口对象，一个页面可绑定多个场景；首期执行时仅选择一个最高优先级场景。
- 场景是执行对象，由组件编排形成，承载字段映射、执行模式、按钮配置、异常处理策略。
- 组件是可复用能力单元，场景通过组件组合表达业务流程；组件能力由平台统一维护版本。
- 字段映射与脚本配置均从属于场景版本，不允许脱离版本单独生效，确保可追溯。
- 发布记录与审计记录贯穿页面、场景、组件的变更链路，用于回溯“谁在何时让哪个配置生效”。
- 规则提示配置从属于规则，不作为独立配置对象存在；规则支持同一页面按分行差异化启停。
- 规则提示配置承载提示内容、展示方式、默认生效范围和时效信息，随规则版本统一发布与回滚。
- 规则状态变更记录属于治理记录，记录某分行或某范围对规则执行的启用、停用、延期动作，不直接修改规则定义与提示内容。
- 规则触发日志属于运行记录，记录某次页面办理过程中规则命中并完成处理的事实；静默、浮窗、外部通知三种类型均需记录，不参与配置发布；考虑到触发量级，规则触发日志写入 Elasticsearch 保存。

#### 3.7.1 核心对象字段建议

以下字段建议用于统一对象边界，避免将规则、提示配置、运行日志、治理审计混为同一类对象。

**规则**
- 标识字段：规则 ID、规则编码、规则名称
- 绑定字段：监控网页 ID
- 逻辑字段：条件集合、条件逻辑（AND/OR）、优先级
- 范围字段：生效范围类型、生效范围内容
- 生命周期字段：状态、生效时间、失效时间、版本号
- 治理字段：是否总行强控、创建人、创建时间、更新人、更新时间

**规则提示配置**
- 标识字段：提示配置 ID、所属规则 ID
- 内容字段：提示类型、标题、正文、处理建议
- 处理字段：处理方式、展示优先级、适用节点
- 范围字段：默认适用范围、责任人、维护分行
- 时效字段：提示生效时间、提示失效时间
- 审计字段：创建时间、更新时间
- 约束说明：规则提示配置不单独持有启停状态，启停统一由规则控制

**规则触发日志**
- 标识字段：日志 ID、规则 ID、命中时规则版本
- 上下文字段：页面 ID、页面 URL、操作人、分行、触发时间
- 结果字段：提示类型、触发原因、处理结果、链路标识
- 明细字段：成功触发时记录当次所有字段匹配值
- 存储说明：规则触发日志存储于单个 Elasticsearch 索引中，日志主体与字段匹配明细合并保存，用于承载高频写入与检索
- 约束说明：规则触发日志只记录运行事实，不承载配置定义与治理状态

**规则状态变更记录**
- 标识字段：记录 ID、规则 ID、操作时规则版本
- 动作字段：操作类型（发布、启用、停用、延期、回滚）
- 操作字段：操作人、操作人分行、操作时间、操作原因
- 影响字段：作用范围、操作前状态、操作后状态
- 约束说明：规则状态变更记录只记录治理动作，不单独保存提示内容

#### 3.7.2 对象关系建议

- 一个规则绑定一个监控网页，可包含多条条件
- 一个规则对应一份规则提示配置
- 一个规则可产生多条规则触发日志
- 一条规则触发日志在单个 Elasticsearch 文档中包含当次字段匹配明细
- 一个规则可产生多条规则状态变更记录

#### 3.7.3 建模约束

- 不保留“提示任务”对象，避免与规则、提示配置、规则触发日志重复建模
- 不保留“提示独立启停”对象，提示是否展示由规则状态决定
- 不在规则提示配置中维护独立启停状态，避免与规则状态形成双真相源
- 条件模型底层允许多个条件标记为可重复触发，但首期产品层仅开放单个条件配置

#### 3.7.4 规则触发日志 Elasticsearch 文档建议

规则触发日志建议按“单次触发一条文档”的方式写入单个 Elasticsearch 索引，文档主体与字段匹配明细合并保存。

**建议主字段**
- 日志标识：日志 ID、链路 ID
- 规则标识：规则 ID、规则编码、规则名称、命中时规则版本
- 页面信息：监控网页 ID、页面 URL、页面编码
- 用户信息：用户 ID、用户姓名、部门 ID、部门名称、分行 ID、分行名称
- 触发信息：提示类型、触发原因、触发时间、处理结果
- 运行信息：客户端时间、上报时间、浏览器标签页标识、插件版本

**字段匹配明细**
- 采用数组结构内嵌在同一文档中
- 每个明细项至少包含：条件 ID、元素 ID、元素逻辑名、选择器、来源类型、实际取值、比较值、是否命中
- 首期仅要求成功触发时写入完整字段匹配明细；失败触发只保留摘要失败原因

**建议查询维度**
- 按规则：规则 ID、规则编码、提示类型
- 按页面：监控网页 ID、页面 URL、页面编码
- 按人群：用户 ID、部门 ID、分行 ID
- 按时间：触发时间范围
- 按结果：处理结果、触发原因

**建议索引约束**
- 单个文档承载一次规则触发的全部信息，不再拆分主体索引与明细索引
- 日志写入以高频追加为主，不支持运行期更新文档内容
- 查询页面默认基于规则 ID、页面、分行、时间范围做组合检索

### 3.8 配置生命周期与发布治理
- 配置生命周期统一为：草稿、待发布、已发布、已下线四个状态。
- 新增或编辑配置默认进入草稿状态，草稿不影响线上执行。
- 配置提交发布后生成新版本并立即生效，保留上一个稳定版本作为回滚基线。
- 回滚操作应以”版本”为最小单位，回滚后需自动记录回滚原因、操作人、时间与影响范围。
- 同一配置对象在同一时刻仅允许一个”已发布”版本，防止运行期读取歧义。
- 发布前应执行配置完整性校验，至少校验页面绑定、组件连接有效性、必填参数、执行模式合法性。
- 对高风险页面支持灰度发布：按用户范围逐步放量；灰度失败可一键回退到稳定版本。
- 首期高风险配置操作（发布/回滚/下线）暂不强制双人复核，采用”先执行 + 全量审计留痕”治理策略。

### 3.9 权限模型与职责分工
- 角色至少包含：配置管理员、业务配置人员、审核人员、只读审计人员。
- 总行管理角色可下发全行强控规则，并可查看全行范围执行与规则状态变更审计。
- 配置管理员负责页面注册、组件能力启停、全局参数与发布策略维护。
- 业务配置人员负责场景编排、字段映射、按钮配置与预览调试，不可直接删除审计记录。
- 审核人员可触发执行、查看执行结果与差异输出，不可修改场景逻辑。
- 只读审计人员仅可查看版本、发布记录、执行日志与字段级审计明细，不可执行发布或回滚。
- 发布、回滚、下线属于高风险操作，必须保留完整审计留痕并可按角色隔离查询。

### 3.10 配置生效与插件运行策略
- 插件进入页面后先执行页面识别，再拉取该页面当前生效配置，未命中时不执行自动流程。
- 插件应支持本地短期缓存最近一次有效配置，用于短时网络异常场景下的可用性保障。
- 插件缓存按页面维度存储，采用localStorage持久化；单页面缓存大小限制为100KB，超出时自动清理最旧缓存。
- 插件缓存粒度按监控网页维度管理；同一页面在不同标签页可复用同一份缓存配置，但规则触发状态与展示状态按标签页隔离。
- 当缓存配置版本落后于中心配置时，插件应在下一次拉取成功后自动切换到最新生效版本。
- 配置发布后，插件侧新版本感知目标为 10 分钟内完成生效。
- 配置拉取失败时，系统需提示”配置不可用”并降级为手工模式，不得阻断用户原流程。
- 场景执行时必须使用单一确定版本，执行中不允许热切换版本，避免同一单据出现混合结果。
- 配置中心下发停用指令后，插件应在下一次拉取周期内停止对应页面自动执行能力。
- 插件与服务端应支持增量拉取机制（本地版本号比对），无更新时返回轻量响应以减少传输与计算开销。
- 插件在短时断连场景可使用本地缓存规则继续运行，恢复连接后自动同步最新版本并覆盖缓存。
- 插件离线缓存连续可用时长上限为 60 分钟；超过上限后必须降级为手工模式并提示用户。
- 插件运行需同时兼容传统多页跳转页面与单页应用（SPA）页面；对 SPA 页面至少需识别首屏加载、路由切换、DOM 结构变化三类事件。

### 3.11 可观测性与运维支持
- 配置中心需提供发布看板，展示近7天发布次数、回滚次数、失败场景与影响页面分布。
- 执行监控至少包括：触发次数、成功率、失败原因分布、平均执行时长、重试成功率。
- 失败原因应按可运营维度分类（页面识别失败、字段缺失、接口异常、脚本异常、权限不足）。
- 对连续失败场景应支持告警与自动熔断，熔断期间仅保留手动触发能力并提示维护人员处理。
- 审计查询应支持按页面、场景、版本、操作人、时间范围检索，满足事后复盘要求。
- 提示监控至少包括：提示触发率、提示阅读确认率、分行停用及时性、风险提示命中后处置结果。

### 3.12 智能提示配置中心设计要求
- 提示标签管理：支持为提示配置一个或多个业务标签，用于表达监管临时要求、事中风险提示、新员工作业指导等业务语义；标签不作为“提示类型”枚举，也不直接参与运行时优先级计算。
- 总行强控管理：支持总行级规则一键下发全行生效；该类规则分行仅可查看，不可停用或修改。
- 提示内容管理：支持标题、正文、适用节点、处理建议、生效时间、失效时间、责任人、维护分行等基础属性。
- 提示处理管理：本期支持静默、浮窗、外部通知三种处理类型；其中浮窗展示不得遮挡核心操作区域，侧边提示与关键字段旁提示纳入后续迭代，外部通知的具体对接细节不在本文展开。
- 提示冲突管理：同一时刻命中多条提示时允许并行生效，并按规则优先级排序处理；浮窗类型进入串行展示队列，静默与外部通知类型不进入浮窗队列；同类重复提示需合并去重。
- 提示触发控制：默认同页面同规则单次触发；若规则中的某个条件被标记为可重复触发，则仅在该条件涉及字段值变化后，该规则才可再次触发。
- 规则时效管理：分行管理人员可手工执行规则启用、停用、延期；所有状态变化必须以规则状态变更记录形式留痕。
- 提示审计管理：可按提示类型、分行、页面、操作人、时间范围检索提示全生命周期记录，至少包含规则触发日志与规则操作记录两类数据。

## 4. 非功能需求

### 4.1 性能要求
- 配置拉取与页面识别引入的页面额外耗时不超过 2000 毫秒，且用户主观感知无明显卡顿。
- 自动流程执行失败率不超过 10%，按天统计，统计分母为“触发次数”。
- 配置发布后，插件侧应在 10 分钟内完成新版本感知并生效。
- 智能提示触发与展示应满足“办理过程内实时可见”要求，不得出现明显延迟导致提示失效。
- 首期成效评估采用双目标：合规风险指标（提示命中率、风险处置率）与效率质量指标（作业耗时、录入差错率）并行考核。

### 4.2 安全要求
- 仅在银行内部授权环境与授权账号下使用。
- 不向未授权外部环境暴露业务敏感数据。
- 关键自动操作需保留审计日志，支持事后追溯操作人、操作时间、操作对象与结果状态。
- 审计日志按“整单+字段明细”记录，字段明细需可追踪关键字段变更。
- 审计日志查看需按角色权限隔离。
- 提示渲染需执行 XSS 白名单过滤，至少允许 `p`、`br`、`strong`、`em`、`ul`、`ol`、`li`、`a(http/https)` 等安全标签。
- 接口定义中的敏感信息（如 API Key、密钥）需加密存储并脱敏展示。
- 自定义预处理脚本沙箱隔离能力首期暂不实现，后续迭代补充安全限制。
- 后端代调接口需启用 SSRF 防护（域名白名单、内网地址限制、回环地址限制）。

### 4.3 兼容性要求
- 兼容银行当前主流浏览器版本与既有插件运行环境，不得要求业务系统进行配套改造。
- 页面规则升级应对历史页面配置保持向后兼容，避免因配置中心升级导致存量场景失效。

## 5. 边界与限制

### 5.1 明确包含（In Scope）
- 【一期交付】公共支撑层核心能力：页面资源管理能力，包括监控网站、监控网页、本页元素配置，页面识别规则维护（页面编码API优先、URL规则兜底），属于两个子系统共享的静态配置资源。
- 【一期交付】智能提示（哨兵）核心能力：运行时页面识别、元素解析、规则匹配能力，产出包含全量页面元素信息的触发事件，智能作业模块无需重复建设页面识别能力，直接复用哨兵模块的识别结果与触发事件。
- 【一期交付】提供哨兵规则核心模型配置能力（规则、规则模板）。
- 【一期交付】提供条件引擎配置能力（左值/右值来源、运算符、预处理链、AND/OR 逻辑）。
- 【一期交付】提供接口定义中心与预处理器中心，支持规则取值扩展与跨规则复用。
- 【一期交付】提供资源共享与权限治理（归属部门、可见性、共享范围、复制复用、引用校验）。
- 【一期交付】建设独立后台配置系统，用于管理配置及版本记录。
- 【一期交付】通过哨兵插件与哨兵后台提供触发条件配置与页面变化事件产出能力。
- 提供智能提示配置能力，支持为提示配置业务标签，用于表达临时监管要求、事中风险提示、新员工作业指导等业务语义。
- 提供智能提示的生命周期管理能力（启用、停用、替代下线），防止临时规则长期滞留。
- 提供智能提示模板与术语规范管理能力，保障提示文案口径一致。
- 提供多提示并行生效能力、提示优先级排序处理能力与提示审计查询能力。
- 在既有浏览器插件基础上实现接口取数并自动注入当前页面。
- 在命中页面绑定时由插件注入悬浮按钮，提供统一执行入口。
- 提供场景级悬浮按钮个性化配置能力，并在命中场景时按场景配置渲染。
- 在后台提供拖拽式复杂注入编排能力，支持业务人员低门槛配置。
- 在后台提供场景编排能力，支持从组件库拖拽组件并连线组成完整场景。
- 提供首期组件能力：录入、赋值、接口请求、获取字段值。
- 建设配置生命周期管理能力（草稿、发布、下线、回滚）与版本发布治理能力。
- 建设角色权限管理与高风险操作审计能力（发布/回滚/下线可追溯）。
- 提供插件侧配置拉取、缓存、失效与降级策略，保障运行连续性。
- 提供配置发布与执行监控看板，支持失败归因与问题定位。
- 提供预览模式，用户确认后才执行自动录入。
- 对外输出页面字段信息，供第三方差异高亮应用使用。
- 在异常情况下给出可理解提示、自动降级与重试能力，并支持用户回退手工处理。
- 智能作业场景执行失败后，系统自动降级为手工模式并在页面显著位置展示"一键重试"按钮；重试操作仅重新执行当前失败场景，不影响其他已完成操作。

### 5.2 明确排除（Out of Scope）
- 替代业务系统原生审批流与最终业务确认动作。
- 对银行核心系统进行后端接口改造。
- 第三方差异高亮应用内部实现逻辑。
- 第三方差异高亮不可用时的本系统差异兜底展示（如纯文本差异清单）首期不纳入交付。
- 智能提示“阅读确认率”运营指标体系与日常跟踪机制（纳入后续迭代）。
- 对未纳入字段清单的全量智能识别与自动决策。
- 同页多场景串行/组合执行。
- 首期组件清单之外的新组件类型。
- 场景回环编排能力。
- 在智能作业后台内重复建设触发条件配置能力、页面识别能力，相关能力完全复用哨兵模块。
- 以智能提示替代正式系统功能建设；智能提示不作为长期替代方案。
- 取代银行统一身份、统一权限、统一审计平台的底层能力建设。
- 首期跨环境多活发布与跨机构级联发布能力。
- 首期自动修复页面结构变更（仅提示与降级，不自动重写规则）。
- 自动生成提示文案或自动判定停用时机（首期由分行管理人员手工维护）。
- 首期 AI 自动识别页面元素、自动生成选择器、自动推荐规则能力。
- 首期规则模板市场化能力（评分、收藏、公开模板广场）。

### 5.3 已知约束
- 当前能力依赖浏览器插件注入机制。
- 目标系统页面结构变化可能影响自动录入稳定性。
- 不得影响作业人员原有系统使用体验。
- 当前第一期重点为“从接口获取数据并注入当前页面”。
- 一期允许仅上线单块能力；当前优先上线智能提示能力。

## 6. 验收标准（Acceptance Criteria）

- **Given** 用户已安装并启用插件，且已登录作业系统并可访问目标页面  
- **When** 用户触发自动录入  
- **Then** 系统展示预览内容，用户确认后目标页面对应字段被正确填充，并展示本次执行状态

- **Given** 页面存在已启用的场景绑定  
- **When** 用户打开目标页面  
- **Then** 插件注入悬浮按钮，并按场景执行模式执行（自动执行、预览确认或手动触发）

- **Given** 管理人员已配置监控网站、监控网页 URL 规则与本页元素  
- **When** 插件进入匹配页面  
- **Then** 系统可基于页面匹配结果加载对应网页规则并正确解析元素取值

- **Given** 哨兵后台已配置触发条件且哨兵插件正常运行  
- **When** 页面变化命中触发条件  
- **Then** 哨兵插件产出触发事件，智能作业插件基于事件判定并执行命中场景

- **Given** 已配置“临时过渡类提示”对应的规则，且由分行管理人员纳入规则治理  
- **When** 正式系统功能上线后分行管理人员确认停用该规则  
- **Then** 对应临时提示不再展示，且保留规则停用审计记录

- **Given** 已配置带有“事中风险”标签的规则  
- **When** 办理过程命中风险触发条件  
- **Then** 页面实时展示风险提示信息并记录规则触发日志

- **Given** 已配置带有“新员工作业指导”标签的规则  
- **When** 新员工进入目标业务页面并触发关键步骤节点  
- **Then** 系统展示标准化操作指引，且不影响正常业务办理流程

- **Given** 同一页面同一时刻命中多条不同规则，且标签不同
- **When** 插件执行提示渲染
- **Then** 多条提示按优先级排序后串行展示（逐条弹窗），用户关闭当前提示后自动展示下一条，且重复提示被合并不重复渲染

- **Given** 同一页面配置了多个场景且优先级不同
- **When** 插件完成页面识别并命中多个场景
- **Then** 首期仅执行最高优先级场景，其余场景不自动执行

- **Given** 同一页面配置了多个场景且优先级数值相同
- **When** 插件完成页面识别并命中多个场景
- **Then** 按场景创建时间倒序排序，执行最新创建的场景

- **Given** 分行管理人员手工停用某条临时提示对应的规则  
- **When** 停用操作提交成功  
- **Then** 该分行对应页面不再展示该规则产生的提示，且可查询完整规则停用审计记录

- **Given** 同一页面已触发某条提示规则且页面状态未发生触发条件变化  
- **When** 用户继续在当前页面办理业务  
- **Then** 该规则不重复触发

- **Given** 某条提示规则中的某个条件已配置为可重复触发  
- **When** 该条件涉及字段的值发生变化并再次满足该规则  
- **Then** 系统将该规则重新放入待触发队列，并允许再次触发且记录新的触发日志

- **Given** 某条提示规则在执行过程中发生接口调用失败、预处理失败或判定超时  
- **When** 系统进行规则判定  
- **Then** 该规则不触发展示，并记录失败原因日志用于排查

- **Given** 规则条件左右值来源包含“接口”且配置了预处理步骤  
- **When** 插件执行该规则  
- **Then** 系统先获取接口值并完成预处理后再比较，比较过程与结果可审计追溯

- **Given** 某部门仅获得他部门共享资源的只读权限  
- **When** 该部门配置人员尝试直接修改共享资源  
- **Then** 系统拒绝编辑；复制为本部门资源后方可修改且不影响原资源

- **Given** 某已发布规则引用的接口或预处理器被删除或权限回收  
- **When** 配置人员尝试再次发布或编辑该规则  
- **Then** 规则被标记为引用失效并禁止发布，直至替换为有效引用

- **Given** 业务配置人员已为场景配置悬浮按钮参数  
- **When** 插件在任一命中页面注入悬浮按钮  
- **Then** 按命中场景的个性化配置展示位置与信息并执行对应交互行为

- **Given** 用户已进入可执行页面且满足自动触发条件  
- **When** 系统判定满足自动触发规则  
- **Then** 自动录入或字段提取流程可被自动触发并输出执行结果

- **Given** 页面字段可被读取且第三方差异应用可用  
- **When** 用户触发差异流程  
- **Then** 系统成功提取页面字段信息并交付第三方差异应用处理

- **Given** 自动流程执行中出现字段缺失或页面异常  
- **When** 系统检测到异常  
- **Then** 系统提示异常原因、自动降级至手工作业并支持一键重试，不出现页面不可用

- **Given** 业务配置人员在后台修改字段配置并提交发布  
- **When** 发布完成  
- **Then** 新配置生效且生成可追溯版本记录

- **Given** 页面配置维护人员已完成页面注册并启用页面规则  
- **When** 插件进入目标页面并执行识别  
- **Then** 系统可命中唯一页面配置并加载对应注入场景

- **Given** 页面已配置页面编码与URL保底规则  
- **When** 插件进入页面执行识别  
- **Then** 系统优先按页面编码命中；页面编码不可用时按URL保底命中

- **Given** 业务配置人员通过拖拽方式完成复杂注入编排  
- **When** 执行预览调试并保存  
- **Then** 可通过组件编排在无需脚本的前提下得到正确注入结果，且配置版本可追溯

- **Given** 业务配置人员进入场景编排页面  
- **When** 从组件库拖出组件并建立连线  
- **Then** 系统可生成可执行的完整场景流程并支持保存

- **Given** 业务配置人员进行场景连线编排  
- **When** 配置包含分支但不包含回环  
- **Then** 系统允许保存并可执行；若存在回环则给出不支持提示

- **Given** 业务配置人员创建新场景  
- **When** 选择可用组件  
- **Then** 系统仅提供首期组件（录入、赋值、接口请求、获取字段值）用于编排

- **Given** 审计人员具备查看权限  
- **When** 查询指定业务单据执行记录  
- **Then** 可查看整单执行记录及字段级明细，且越权角色不可见

- **Given** 业务配置人员完成场景编辑但未发布  
- **When** 审核人员进入目标页面执行  
- **Then** 系统仍使用上一个已发布版本，不读取草稿内容

- **Given** 配置管理员发布新版本后出现异常  
- **When** 执行回滚操作  
- **Then** 系统按版本回退到最近稳定版本并记录回滚原因、操作人、时间与影响范围

- **Given** 配置中心与插件网络短时异常  
- **When** 插件无法实时拉取最新配置  
- **Then** 系统可基于本地有效缓存继续可控运行，若缓存不可用则降级为手工模式并提示用户

- **Given** 配置发布已完成  
- **When** 插件按拉取周期同步配置  
- **Then** 插件应在 10 分钟内感知并生效新版本，超时应记录告警日志

- **Given** 插件与配置中心持续断连  
- **When** 离线缓存运行时长超过 60 分钟  
- **Then** 系统停止自动执行并降级为手工模式，同时提示当前配置可能已过期

- **Given** 同一页面存在多个场景且配置了优先级
- **When** 插件完成页面识别并命中多个场景
- **Then** 首期仅执行最高优先级场景，其余场景不自动执行

- **Given** 管理员执行高风险操作（发布/下线/回滚）
- **When** 审计人员按时间范围检索
- **Then** 可查询完整操作轨迹，包含操作前后版本信息与操作结果

- **Given** 配置人员使用规则模板创建规则但未完成所有占位元素映射
- **When** 尝试保存规则配置
- **Then** 系统禁止保存并提示"模板映射不完整，请为所有占位元素指定目标页面元素"

- **Given** 规则条件中引用的预处理器执行失败
- **When** 插件执行该规则判定
- **Then** 整个规则判定失败，按"失败不触发"策略处理并记录失败日志

- **Given** 智能作业场景执行过程中发生异常失败
- **When** 系统检测到执行失败
- **Then** 自动降级为手工模式并在页面显著位置展示"一键重试"按钮，用户点击后重新执行当前失败场景

- **Given** 页面元素被规则或场景引用后，配置人员修改该元素并发布
- **When** 元素修改完成
- **Then** 系统生成新版本，引用方可自行选择是否升级到新版本（升级提示机制后期迭代实现）
