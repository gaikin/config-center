# 营小助配置中心实施计划评审报告

> 关联 Spec：spec/smart-operation/spec.md
> 关联 Plan：spec/smart-operation/plan.md
> 评审日期：2026-03-10
> 评审范围：实施计划文档评审

## 评审总结

当前实施计划整体覆盖面较完整，主链路、任务拆解和测试场景已经具备较好的可执行性。未发现会直接阻断实施的 P0 级问题，但存在多项 P1 级问题需要在进入研发前收口，否则容易在对象建模、发布治理、审计追溯和权限治理上出现返工。另有少量 P2 级结构性问题，建议在定版前一并清理。

## P1 - 建议修复（重要但不阻塞）

### [P1-1] 规则匹配执行位置存在冲突
- **类型**：架构问题
- **位置**：`spec/smart-operation/plan.md:63`、`spec/smart-operation/plan.md:382`、`spec/smart-operation/plan.md:464`
- **描述**：计划中同时存在“JavaScript 规则引擎在插件侧执行”和“服务端执行规则匹配逻辑”两套口径，后续很容易演变为双实现、双验收标准。
- **影响**：会导致规则判定逻辑、失败策略、调试口径不一致，增加实现和测试复杂度。
- **建议**：统一为“服务端负责配置过滤和接口代理，插件负责最终规则匹配与触发”。

### [P1-2] 资源共享模型未完整落地
- **类型**：功能缺失
- **位置**：`spec/smart-operation/plan.md:331`
- **描述**：当前计划主要体现了元素共享（只读复用、Fork），但 spec 要求的是统一资源共享模型，至少应覆盖归属部门、可见性、共享范围，以及规则引用资源的统一权限治理。
- **影响**：接口、预处理器、模板等资源无法使用统一共享与校验模型，规则发布时也无法完整判断权限和引用有效性。
- **建议**：补充统一资源共享治理能力，并将引用资源的权限校验纳入规则发布链路。

### [P1-3] 下发范围未体现按用户维度控制
- **类型**：功能缺失
- **位置**：`spec/smart-operation/plan.md:391`
- **描述**：已确认一期不单独建设灰度发布模块，而是通过规则/策略下发范围实现放量控制；但当前计划中的生效范围仅体现部门及子部门，未明确支持按用户范围下发。
- **影响**：无法承接 spec 中“按用户范围逐步放量”的要求。
- **建议**：在规则提示配置相关任务中明确支持按用户范围下发，回退仍复用版本回滚能力。

### [P1-4] 接口定义敏感信息安全要求缺失
- **类型**：安全问题
- **位置**：`spec/smart-operation/plan.md:360`、`spec/smart-operation/plan.md:549`
- **描述**：计划中未体现 API Key、密钥等敏感字段的加密存储、脱敏展示、编辑不回显等安全要求。
- **影响**：存在明文存储或前端直接暴露敏感凭据的风险，不满足 spec 安全要求。
- **建议**：在接口定义管理后端和前端任务中补充敏感字段安全口径，并在测试标准中增加对应校验。

### [P1-5] 缺少发布看板与执行监控任务
- **类型**：功能缺失
- **位置**：`spec/smart-operation/plan.md:591`
- **描述**：当前计划已有审计查询，但未落地 spec 要求的发布看板和执行监控能力。
- **影响**：上线后只能查日志，无法满足发布统计、失败归因和问题定位的日常运维诉求。
- **建议**：新增轻量版发布看板/执行监控任务，至少覆盖发布次数、回滚次数、失败原因分布、触发次数、成功率、平均执行时长等基础指标。

### [P1-6] 规则触发日志缺少成功触发字段明细与高频存储方案
- **类型**：功能缺失
- **位置**：`spec/smart-operation/plan.md:426`、`spec/smart-operation/plan.md:491`
- **描述**：当前日志设计仅覆盖提示类型、触发原因、页面、操作人、分行等摘要信息。评审中已确认：成功触发时需记录所有字段的匹配值；失败触发首期可不记录字段级明细。同时考虑到高频写入与多条件查询，规则触发日志不宜继续按普通业务表方式设计。
- **影响**：后续无法从审计日志中追溯“哪些字段值导致了本次命中”，排障和复盘成本高。
- **建议**：补充规则触发日志字段明细；失败场景首期仅保留摘要失败原因。规则启停相关审计仍归入规则操作记录，不单独建“提示启停”对象。考虑触发量级，规则触发日志建议写入 Elasticsearch 保存并用于检索，日志主体与字段明细可合并到同一个索引文档中。

### [P1-7] 发布前引用校验与引用失效禁止发布未写实
- **类型**：逻辑缺失
- **位置**：`spec/smart-operation/plan.md:375`、`spec/smart-operation/plan.md:564`
- **描述**：计划中规则 CRUD 和规则配置页面未明确体现“发布前校验引用资源权限与完整性”“引用失效时禁止发布”的行为。
- **影响**：可能出现引用资源被删除、权限回收后仍可继续发布规则的情况，运行时才暴露错误。
- **建议**：在规则发布链路中补充引用完整性校验、引用资源权限校验、引用失效标记，以及前端阻断发布提示。

### [P1-8] 规则与规则提示配置的关系表述不清
- **类型**：设计表达问题
- **位置**：`spec/smart-operation/plan.md:374`、`spec/smart-operation/plan.md:390`、`spec/smart-operation/plan.md:419`、`spec/smart-operation/plan.md:577`
- **描述**：根据评审确认，提示配置应依附规则存在，而不是独立于规则并行生效的第二套运行时对象。但当前计划文案容易让人理解为“规则”和“规则提示配置”是两套并列对象。
- **影响**：容易导致后续表结构、接口返回和前端编辑模型出现双真相源。
- **建议**：统一文案为“规则主对象 + 附属提示配置”，避免并列描述。

## P2 - 可选优化（锦上添花）

### [P2-1] 依赖关系与并行组存在重复和不一致
- **类型**：文档优化
- **位置**：`spec/smart-operation/plan.md:698`、`spec/smart-operation/plan.md:699`、`spec/smart-operation/plan.md:704`、`spec/smart-operation/plan.md:705`
- **描述**：并行组中存在 `TODO-S5`、`TODO-P4` 重复出现，且不同组的粒度和表达不完全一致。
- **影响**：不影响方案正确性，但会增加排期沟通和执行理解成本。
- **建议**：统一清理依赖关系和并行组描述，保证每个任务只在一个合理分组中出现。

## 已确认的范围收敛

- 提示模板与术语规范管理：一期暂不处理，由分行配置人员自由维护提示文案。
- 灰度发布：不新增独立灰度发布模块，通过规则/策略下发范围控制实现。
- 审计明细：成功触发记录所有字段匹配值；失败触发首期仅保留摘要失败原因，不记录字段级明细。

## 建议的计划修订方向

1. 将 `TODO-R6` 收敛为配置过滤/接口代理相关能力，不再描述为服务端完整规则匹配。
2. 在 `TODO-S1` 中明确支持按用户范围下发。
3. 在 `TODO-R3`、`TODO-F7` 中补充敏感信息加密存储与脱敏展示要求。
4. 补充发布看板与执行监控相关后端统计接口和前端页面任务。
5. 在 `TODO-S6`、`TODO-E9` 中补充成功触发字段明细记录要求。
6. 在 `TODO-R5`、`TODO-F9` 中补充引用校验、引用失效标记和禁止发布逻辑。
7. 统一“规则主对象 + 附属提示配置”的表述。
8. 清理依赖关系与并行组中的重复项。
